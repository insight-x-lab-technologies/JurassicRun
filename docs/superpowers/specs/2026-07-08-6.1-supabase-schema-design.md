# 6.1 — Schema Supabase (design)

> Fase 6 (Online — Supabase), item 6.1. **Só DDL/infra — não toca `src/core/`** ⇒ determinismo
> 67 inalterado. Entrega os artefatos de schema; a aplicação no banco vivo é passo do usuário.

## Objetivo

Definir o schema do banco (tabelas + RLS) que hospeda os recursos online do JurassicRun,
isolado num banco Supabase **compartilhado entre projetos** (`InsightXLabGamesHub`). Este item
entrega **apenas o schema e seus artefatos versionados** — o wiring de cliente/serviços é 6.2–6.5.

## Contexto e restrições

- Banco Supabase compartilhado por vários jogos ⇒ **isolamento obrigatório** (pedido do usuário:
  "prefixo JR ou outra técnica").
- `RLS adequada` é requisito explícito do roadmap (6.1).
- Modelos de dados já existem localmente (Fases 4–5); o schema os espelha para permitir o
  espelhamento central em 6.3/6.5 e a verificação em 6.4:
  - `Profile {id, name, createdAt}` — `src/services/profile/store.ts`.
  - `LeaderboardResult {mode, seed, score, distance, food, nearMisses, level, achievedAt}` —
    `src/services/leaderboard/store.ts`. `score` é **float canônico** (economia 1.8).
  - `StoredReplay {mode, seed, timeline: boolean[], score, distance, food, nearMisses,
    finalHash, achievedAt}` — `src/services/replay/store.ts` (5.4).
  - Troféus: ids de catálogo (`firstFlight … dailyPodium`) + `TrophyStats` —
    `src/services/trophy/`.
- **Credenciais disponíveis:** Project URL + publishable (anon) key. **Ausentes:** senha do
  Postgres e `service_role` key ⇒ este agente **não pode** aplicar a migração no banco vivo.

## Decisões técnicas (autônomas)

### D1 — Isolamento por schema Postgres dedicado `jurassicrun`
Em vez de prefixar cada objeto com `jr_` no schema `public`, todos os objetos do JurassicRun
vivem num schema dedicado `jurassicrun`. Motivo: num banco multi-jogo, um schema é o mecanismo
idiomático de isolamento — zero colisão de **tabelas, tipos, funções, triggers e nomes de
policy** (o prefixo teria de ser repetido em todos esses; o schema resolve de uma vez). Supera o
prefixo mantendo o mesmo objetivo. **Passo de config no Supabase:** adicionar `jurassicrun` a
_Exposed schemas_ (Settings → API) para o PostgREST/anon key enxergar; o cliente JS usará
`createClient(url, key, { db: { schema: 'jurassicrun' } })` (wiring em 6.2).

### D2 — Identidade via Supabase Auth anônimo; `players.id = auth.uid()`
Cada dispositivo faz _anonymous sign-in_ (6.2) ⇒ ganha um `auth.users.id` (uuid) = **ID global
único** do jogador (satisfaz 6.2). A PK de `players` referencia `auth.users(id)`. Isso habilita
RLS por linha real (`... = auth.uid()`) — a base do anti-cheat. Alternativa rejeitada: uuid
gerado no cliente + RLS permissiva (qualquer um insere score com qualquer `player_id`) — falsa
segurança, incompatível com "RLS adequada" + verificação 6.4.

### D3 — `verified` travado ao `service_role`
`scores` e `challenge_entries` têm `verified boolean not null default false`. Um trigger
`BEFORE INSERT OR UPDATE` **força `verified = false`** quando o autor não é `service_role`
(`auth.role() <> 'service_role'`). Só a Edge Function de verificação (6.4, roda com service_role)
pode marcar `verified = true`. Assim o cliente nunca se auto-verifica — seam do anti-cheat pronto
no schema.

### D4 — Tipos numéricos
`score`, `distance` = `double precision` (espelham o float64 canônico do core). Contadores
inteiros (`food`, `near_misses`, `level`) = `integer` com `check (>= 0)`.

### D5 — `timeline` como `jsonb`
A `InputTimeline` do replay (`boolean[]`) é gravada em `challenge_entries.timeline` como `jsonb`
(array de booleanos). Empacotamento compacto (bitset/base64) fica p/ backlog (já adiado no 5.4);
`jsonb` mantém a paridade 1:1 com `StoredReplay` e é o que a Edge Function 6.4 vai reidratar.

## Tabelas (schema `jurassicrun`)

Todas com **RLS habilitada**. `created_at timestamptz not null default now()`.

### `players`
| coluna | tipo | notas |
|---|---|---|
| `id` | `uuid` PK | `references auth.users(id) on delete cascade` |
| `name` | `text not null` | `check (char_length(name) between 1 and 20)` (espelha `NAME_MAX=20`) |
| `avatar` | `text` | nullable; cosmético derivado no cliente hoje, coluna p/ futuro |
| `created_at` | `timestamptz` | |

**RLS:** SELECT público (leaderboards exibem nomes); INSERT/UPDATE/DELETE só onde `id = auth.uid()`.

### `scores` — entradas de leaderboard (todos os modos)
| coluna | tipo | notas |
|---|---|---|
| `id` | `bigint` PK | `generated always as identity` |
| `player_id` | `uuid not null` | `references players(id) on delete cascade` |
| `mode` | `text not null` | `check (mode in ('endless','daily','weekly'))` |
| `seed` | `text not null` | |
| `score` | `double precision not null` | |
| `distance` | `double precision not null` | |
| `food` | `integer not null` | `check (>= 0)` |
| `near_misses` | `integer not null` | `check (>= 0)` |
| `level` | `integer not null` | `check (>= 0)` |
| `verified` | `boolean not null default false` | travado ao service_role (D3) |
| `created_at` | `timestamptz` | |

**RLS:** SELECT público; INSERT só onde `player_id = auth.uid()`; **sem UPDATE/DELETE** pelo
cliente (scores são imutáveis; `verified` vira true só via service_role, que ignora RLS).

### `challenge_entries` — replays verificáveis de desafio (Diário/Semanal)
| coluna | tipo | notas |
|---|---|---|
| `id` | `bigint` PK | `generated always as identity` |
| `player_id` | `uuid not null` | `references players(id) on delete cascade` |
| `mode` | `text not null` | `check (mode in ('daily','weekly'))` |
| `seed` | `text not null` | |
| `score` | `double precision not null` | |
| `distance` | `double precision not null` | |
| `food` | `integer not null` | `check (>= 0)` |
| `near_misses` | `integer not null` | `check (>= 0)` |
| `timeline` | `jsonb not null` | array de booleanos (flap por step) |
| `final_hash` | `text not null` | `check (final_hash <> '')` — âncora que 6.4 recomputa |
| `verified` | `boolean not null default false` | travado ao service_role (D3) |
| `created_at` | `timestamptz` | |
| | | `unique (player_id, seed)` — 1 melhor tentativa por período/jogador |

**RLS:** SELECT público; INSERT/UPDATE só onde `player_id = auth.uid()` (UPDATE para melhorar a
melhor tentativa; o trigger D3 reseta `verified` no update do dono).

### `trophies` — conquistas por jogador
| coluna | tipo | notas |
|---|---|---|
| `player_id` | `uuid not null` | `references players(id) on delete cascade` |
| `trophy_id` | `text not null` | id do catálogo (`firstFlight … dailyPodium`) |
| `unlocked_at` | `timestamptz not null default now()` | |
| | | PK `(player_id, trophy_id)` |

**RLS:** SELECT público (perfis mostram troféus); INSERT só onde `player_id = auth.uid()`.
Troféus centrais (top-3 diário, 6.5) são concedidos via service_role (ignora RLS).

## Trigger de integridade (D3)

```sql
create or replace function jurassicrun.lock_verified()
returns trigger language plpgsql as $$
begin
  if auth.role() is distinct from 'service_role' then
    new.verified := false;
  end if;
  return new;
end $$;
```
Anexado `before insert or update` em `scores` e `challenge_entries`.

## Artefatos entregues

1. `supabase/migrations/<timestamp>_jr_schema.sql` — schema + tabelas + RLS + policies + trigger,
   **idempotente** (`create schema if not exists`, `drop policy if exists` antes de recriar).
2. `supabase/README.md` — como aplicar (SQL Editor **ou** `supabase db push`), passo _Exposed
   schemas_, e o passo manual de _Enable anonymous sign-ins_ (Auth settings) que o 6.2 exige.
3. `.env.example` — `VITE_SUPABASE_URL` + `VITE_SUPABASE_ANON_KEY` (publishable key), com
   comentário de que `.env` real é git-ignored. Exceção `!.env.example` no `.gitignore`.
4. `src/services/online/schema.ts` — módulo puro de **constantes compartilhadas**:
   `SUPABASE_SCHEMA = 'jurassicrun'`, nomes de tabela (`TABLES.players` etc.), listas de
   modos/colunas usadas pela guarda. Sem IO, sem `@supabase` (dependência do cliente é 6.2).
5. `tests/online/schema-contract.test.ts` — **guarda de regressão** (molde das guardas de
   determinismo/i18n): lê o `.sql` da migração e afirma invariantes — cada tabela existe no
   schema `jurassicrun`, cada uma tem `enable row level security`, cada uma tem policy de SELECT
   e de INSERT, o trigger `lock_verified` existe e está anexado às 2 tabelas com `verified`, e os
   nomes de tabela/colunas batem com `src/services/online/schema.ts`. Impede que o SQL e as
   constantes TS divirjam silenciosamente.
